# Migration Guide: v0.2 — Content-Metadata Bifurcation

This release adds content-metadata bifurcation support to zodal. Most changes are additive (new types, new functions, new optional features). There is **one breaking change** that may require action.

## Breaking Change

### `ResolvedFieldAffordance` now requires `storageRole`

**What changed.** The `ResolvedFieldAffordance` type in `@zodal/core` now includes a required `storageRole: FieldStorageRole` property (value is `'metadata'` or `'content'`).

**Who is affected.** Code that constructs `ResolvedFieldAffordance` objects manually — typically renderer testers, custom generators, or test fixtures. If you only *consume* `ResolvedFieldAffordance` from `defineCollection` or the UI generators, you are not affected (the engine populates `storageRole` automatically).

**Diagnosis.** After upgrading, run `pnpm build` (or `tsc --noEmit`). TypeScript will flag any object literal assigned to `ResolvedFieldAffordance` that is missing `storageRole`. Common error:

```
Property 'storageRole' is missing in type '{ sortable: ...; filterable: ...; ... }'
  but required in type 'ResolvedFieldAffordance'.
```

**Fix.** Add `storageRole: 'metadata'` to any manually constructed `ResolvedFieldAffordance`:

```typescript
// Before
const field: ResolvedFieldAffordance = {
  sortable: 'both',
  filterable: 'search',
  searchable: true,
  groupable: false,
  editable: true,
  visible: true,
  title: 'Name',
  zodType: 'string',
  zodDef: {},
};

// After — add storageRole
const field: ResolvedFieldAffordance = {
  sortable: 'both',
  filterable: 'search',
  searchable: true,
  groupable: false,
  editable: true,
  visible: true,
  title: 'Name',
  zodType: 'string',
  zodDef: {},
  storageRole: 'metadata',  // ← add this
};
```

### `CollectionState<T>` now includes `contentLoading`

**What changed.** `CollectionState<T>` has a new required property `contentLoading: Record<string, Record<string, boolean>>`.

**Who is affected.** Code that constructs `CollectionState<T>` objects manually (test fixtures, state initialization). If you use `createCollectionStore()` or `createZustandStoreSlice()`, the property is initialized automatically.

**Fix.** Add `contentLoading: {}` to any manually constructed state objects.

### `CollectionActions<T>` now includes `setContentLoading`

**What changed.** `CollectionActions<T>` has a new required method `setContentLoading(state, itemId, field, loading)`.

**Who is affected.** Code that implements `CollectionActions<T>` manually. If you use `createCollectionStore()`, this is handled automatically.

**Fix.** Add the method to your implementation:
```typescript
setContentLoading(state, itemId, field, loading) {
  const itemLoading = { ...state.contentLoading[itemId], [field]: loading };
  return { ...state, contentLoading: { ...state.contentLoading, [itemId]: itemLoading } };
},
```

## New Features

### Content-Metadata Bifurcation (all packages)

**Core (`@zodal/core`):**
- `FieldStorageRole` type: `'metadata' | 'content'`
- `ContentRef` interface: reference to content stored separately
- `isContentRef()` type guard
- `storageRole` property on `FieldAffordance`
- `content` config on `CollectionConfig` (`listStrategy`, `detailStrategy`)
- `getContentFields()`, `getMetadataFields()`, `hasBifurcation()` on `CollectionDefinition`
- Inference engine: fields named `attachment`, `file`, `blob`, `upload`, `media`, `raw_data`, etc. are automatically classified as content

**Store (`@zodal/store`):**
- `createBifurcatedProvider()` — composes two providers (metadata + content) into one unified `DataProvider<T>`
- `splitFields()` — utility to split an object into metadata and content portions
- `getContent()` / `setContent()` — optional methods on `DataProvider<T>` for explicit content access
- `bifurcated` / `contentFields` — new optional properties on `ProviderCapabilities`

**UI (`@zodal/ui`):**
- `ColumnConfig.meta.storageRole` / `meta.isContentRef` — content-aware rendering metadata
- `FormFieldConfig.isContentField` / `acceptMimeTypes` / `maxSize` — file upload config
- `storageRoleIs()` — new composable renderer tester predicate
- `contentLoading` state + `setContentLoading` action — per-item content loading tracking
- Content fields infer `type: 'file'` in form config

### Quick Start: Using Bifurcation

```typescript
import { z } from 'zod';
import { defineCollection } from '@zodal/core';
import { createBifurcatedProvider, createInMemoryProvider } from '@zodal/store';

const DocSchema = z.object({
  id: z.string().uuid(),
  title: z.string(),
  attachment: z.any(),  // auto-classified as content
});

const collection = defineCollection(DocSchema);
console.log(collection.getContentFields());  // ['attachment']
console.log(collection.hasBifurcation());    // true

const provider = createBifurcatedProvider({
  metadataProvider: createInMemoryProvider([]),
  contentProvider: createInMemoryProvider([]),
  contentFields: collection.getContentFields(),
});

// Create an item — metadata and content are routed to different stores
await provider.create({ id: '1', title: 'Doc', attachment: Buffer.from('hello') });

// List items — content fields appear as ContentRef, not actual bytes
const { data } = await provider.getList({});
// data[0].attachment → { _tag: 'ContentRef', field: 'attachment', itemId: '1' }

// Fetch content explicitly
const content = await provider.getContent!('1', 'attachment');
```

## Satellite Package Compatibility

Existing satellite packages (`zodal-store-fs`, `zodal-store-s3`, `zodal-store-supabase`, `zodal-store-localstorage`, `zodal-ui-vanilla`, `zodal-ui-shadcn`) are **not affected** by these changes. They can be used as-is as either the metadata or content provider in a `createBifurcatedProvider` composition.
